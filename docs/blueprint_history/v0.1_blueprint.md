## 1. Concept

This game combines elements of **Memory** and **City–Country–River** into a structured card-matching game.

Each logical set consists of **three linked cards**:

- **Capital**
- **Country**
- **River**

Example:

- Paris
- France
- Loire

A successful match is not a pair, but a **triple**.

The game should support two modes:

1. **Same Country Mode** — three cards belong to the same country.
2. **Same Letter Mode** — three cards are grouped by the same initial letter.

The first implementation is intended as a **local multiplayer game on one screen**, with players taking turns.

---

## 2. Core Design Goal

The game should be:

- easy to understand,
- visually clear on a single monitor,
- educational without feeling like a quiz,
- flexible enough to support curated datasets and later expansion.

The primary design principle is:

> The data model must be clean enough that gameplay remains unambiguous.

---

## 3. Modes

### 3.1 Same Country Mode

**Rule:** a valid set consists of the capital of a country, the country name, and a designated river associated with that country.

**Example:** Vienna · Austria · Danube

**Matching logic:** a triple is correct if all three cards share the same `country_id`.

**Advantage:** semantically strong and educational.

**Challenge:** the river must be curated carefully. "Most important river" is not always objectively defined.

### 3.2 Same Letter Mode

**Rule:** a valid set consists of three cards whose labels begin with the same initial letter.

**Example:** Berlin · Belgium · Brahmaputra

**Matching logic:** a triple is correct if all three cards share the same `initial_letter`.

**Advantage:** simpler and often faster.

**Challenge:** less geographically rigorous unless the dataset is curated for balanced gameplay.

**Recommendation:** Same Letter Mode should use a pre-built curated deck, not arbitrary automatic combinations.

---

## 4. Recommended First Version

For the first playable prototype:

- implement both modes,
- start with a small curated dataset,
- keep the number of cards limited,
- prioritise clarity over scale.

Recommended initial board sizes:

- 12 cards = 4 triples
- 18 cards = 6 triples
- 24 cards = 8 triples

For the very first test build, 18 cards is a good starting point.

---

## 5. Player Experience

### 5.1 Turn Sequence

A player may reveal cards one by one, up to three cards per turn.

**Option A — Strict Triple Turn:**
- Reveal card 1, 2, 3.
- System checks whether the three cards form a valid triple.
- Correct: cards remain open; player scores one point; player continues or ends turn depending on rule setting.
- Incorrect: cards flip back after a short delay; next player takes over.

**Option B — Early Failure Logic:**
- After card 2, the system may detect that no valid triple is possible.
- Turn ends immediately.

**Recommendation:** Use Option A for the first version. It is simpler, more transparent, and easier for players to understand.

### 5.2 Scoring

Base: 1 point for each correct triple.

Optional later additions: bonus for consecutive correct triples, difficulty-weighted scoring, timed mode bonus, penalty for failed attempts.

For version 0.1: simple scoring, one triple = one point.

### 5.3 End of Game

The game ends when all triples have been found. Winner is the player with the highest score. Ties: shared win, or tie-break round in a later version.

## 6. Data Model

The game should separate **semantic objects** from **display cards**.

### 6.1 Entity Model

```json
{
  "country_id": "AUT",
  "country_name": "Austria",
  "capital_name": "Vienna",
  "river_name": "Danube",
  "selection_reason": "major_historic",
  "difficulty": 1,
  "region": "Europe",
  "is_active": true
}
```

### 6.2 Card Model

```json
{
  "card_id": "card_001",
  "country_id": "AUT",
  "mode_tags": ["same_country", "same_letter"],
  "card_type": "capital",
  "label": "Vienna",
  "initial_letter": "V"
}
```

### 6.3 Suggested Fields

**Country-level:** `country_id`, `country_name`, `capital_name`, `river_name`, `selection_reason`, `difficulty`, `region`, `is_active`.

**Card-level:** `card_id`, `country_id`, `card_type`, `label`, `initial_letter`, `mode_tags`.

---

## 7. River Selection Policy

A river should not be chosen arbitrarily. Each country entry should include a justification tag.

**Suggested `selection_reason` values:** `longest_in_country`, `largest_basin`, `capital_region`, `major_historic`, `iconic`, `cross_border_reference`, `none`.

**Important rule:** the game should use one canonical river per country entry, even if alternatives exist, to keep gameplay unambiguous.

---

## 8. Dataset Strategy

- **Phase 1 — Manual Curated Core:** 20 countries and 20 letter triples.
- **Phase 2 — Expanded Database:** more countries, refined letter balance, difficulty levels.
- **Phase 3 — Full Library:** global deck, regional decks, children's mode, expert mode, language switching.

---

## 9. Same Letter Mode Curation Strategy

Same Letter Mode should use curated playable triples, not random combinations.

```json
{
  "letter_group_id": "B_01",
  "initial_letter": "B",
  "cards": [
    { "card_type": "capital", "label": "Berlin" },
    { "card_type": "country", "label": "Belgium" },
    { "card_type": "river", "label": "Brahmaputra" }
  ],
  "difficulty": 2,
  "is_active": true
}
```

---

## 10. Gameplay Architecture

Browser-based local game on one screen. Stack: HTML/CSS/JS (React optional). JSON files. In-browser state.

Game flow: start → mode select → player count → board → play → end → restart.

---

## 11. UI Principles

Large text, strong hidden/revealed distinction, visible current player / score / mode. Cards show only label; optional type hint in advanced modes. For v0.1, hiding type may make the game more interesting.

---

## 12. Difficulty System

Levels 1 (easy), 2 (medium), 3 (hard). Enables children's, school, and expert decks.

---

## 13. Multiplayer Rules

2–4 local players, turn-based, one shared screen.

---

## 14. Risks and Edge Cases

- Country Mode ambiguity: not every country has one obvious "signature river".
- Letter Mode imbalance: some letters are weak or obscure.
- Educational tension: too much rigour reduces fun; too much looseness reduces trust.

Mitigation: separate gameplay deck from reference database.

---

## 15. Minimal Viable Product (MVP)

Includes: mode selection, 2-player local, 6 triples per round, score, turn handling, card flip, end screen.

Not yet: online multiplayer, accounts, database server, full dataset, timers, sound.

## 18. Design Decision Summary

**Keep:** triple-based matching, two modes, curated datasets, local multiplayer first, browser-based.

**Avoid:** full geography automation, global dataset too early, mixed strict-geography/letter logic.

---

## 19. Recommended Next Steps

1. Define first curated dataset (20 country triples, 20 letter triples).
2. Finalise rule wording (turn logic, scoring, card type visibility).
3. Build a simple JSON schema.
4. Implement a local browser prototype.
5. Playtest and revise before scaling.

---

## 20. Short Position

This concept is strong because it combines memory, categorisation, semantic linking, and light geography learning. The key to success is not technical complexity, but clean curation. If the data layer is well designed, the web implementation is straightforward.
