## 1. Concept

This game combines elements of **Memory** and **City–Country–River** into a structured card-matching game.

Each logical set consists of **three linked cards**:

- **Capital**
- **Country**
- **River**

Example:

- Paris
- France
- Loire

A successful match is not a pair, but a **triple**.

The game supports two modes:

1. **Same Country Mode** — three cards belong to the same country.
2. **Same Letter Mode** — three cards share the same initial letter.

The first implementation is a **local multiplayer game on one screen**, with players taking turns.

---

## 2. Core Design Goal

The game should be:

- easy to understand,
- visually clear on a single monitor,
- educationally transparent — players should always be able to tell what kind of entity a card represents by default,
- flexible enough to support curated datasets and later expansion.

The primary design principle is:

> The data model must be clean enough that gameplay remains unambiguous, and the curation must be honest enough that players can trust what they are learning.

A secondary principle:

> Opacity is a difficulty setting, not a default. Hiding card type belongs in an expert mode, not the base experience.

A tertiary principle added in v0.3, following the external critique:

> Governance serves play. Curation metadata, provenance, and dispute channels must remain available but never visually dominate the playing surface.

---

## 3. Modes

### 3.1 Same Country Mode

**Rule:** a valid set consists of the capital of a country, the country name, and a designated river associated with that country.

**Example:** Vienna · Austria · Danube

**Matching logic:** a triple is correct if all three cards share the same `country_id`.

### 3.2 Same Letter Mode

**Rule:** a valid set consists of three cards whose labels begin with the same initial letter, **and** cover the three entity types (one Capital, one Country, one River).

**Example:** Berlin · Belgium · Brahmaputra

**Matching logic:** a triple is correct if all three cards share the same `initial_letter` **and** the triple contains exactly one card of each `card_type`.

### 3.3 Mode Exclusivity

Within a single round, only one mode is active. Cards on the board are drawn from one curated deck (Same Country or Same Letter), never mixed. A triple that would coincidentally satisfy both modes is not possible within a single round, because the active deck governs which triples exist.

### 3.4 Mode Asymmetry [new, v0.3]

The two modes are not educationally equivalent.

- **Same Country Mode** teaches a **semantic relation**: capitals and rivers belong to countries. It exercises geographic memory.
- **Same Letter Mode** teaches a **formal lexical relation**: three labels sharing an initial letter. It exercises phonetic categorisation and pattern memory.

This is not a flaw — the asymmetry is intentional and the modes complement each other — but it must be named. Same Letter Mode is the **lighter, faster, more playful** mode; Same Country Mode is the **slower, more reference-weight** mode. A classroom session might begin with one and end with the other; a family evening might prefer Letter Mode throughout.

The Curation Statement (§18) and any teacher-facing material should state this asymmetry openly to prevent conceptual drift — specifically, to prevent Letter Mode being misread as a geography-reference mode that it is not.

## 4. Recommended First Version

For the first playable prototype:

- implement **both modes**,
- start with a **small curated dataset**,
- keep the number of cards limited,
- prioritise clarity over scale.

Recommended initial board sizes:

- **12 cards** = 4 triples (children or introductory play)
- **18 cards** = 6 triples (**default MVP**)
- **24 cards** = 8 triples (standard play)

These board sizes refer to the **number of cards drawn per round** from a larger curated pool. The Phase 1 pool is 20 triples per mode (§8.1), from which 4–8 triples are selected per round.

### 4a. Pedagogical Progression [new, v0.3]

The user journey is staged:

1. **Tutorial** (§10.4) — three-step introduction on first launch.
2. **Introductory** — Same Country Mode, difficulty Level 1, 12 cards (4 triples).
3. **Standard** — either mode, difficulty Level 2, 18 cards (6 triples). MVP default.
4. **Advanced** — either mode, difficulty Level 3, 24 cards (8 triples).
5. **Expert** — either mode, difficulty Level 4 (type-hidden), 18 or 24 cards.

The menu presents these as a progression, not as equivalent options. A first-time player is guided towards Tutorial → Introductory; a returning player may enter at any stage.

This progression replaces the flat difficulty table of v0.2 as the principal user-facing design, with the difficulty table (§12) retained as the underlying parameter map.

## 5. Player Experience

### 5.1 Turn Sequence

A player reveals cards one by one, up to three cards per turn.

#### Option A — Strict Triple Turn (MVP)

- Reveal card 1.
- Reveal card 2.
- Reveal card 3.
- System checks whether the three cards form a valid triple.

If correct: cards remain open; player scores one point; player's turn ends (see §5.1.1).

If incorrect: cards flip back after a short delay; next player takes over.

#### 5.1.1 Turn Continuation Rule

MVP rule: after a correct triple, the player's turn **ends**. Play passes to the next player regardless of outcome.

Rationale: predictable session length, continuous engagement, and pedagogical symmetry with classical Memory.

### 5.2 Scoring

Base scoring: **1 point** per correct triple.

Optional later additions (not in MVP): consecutive-triple bonus, difficulty-weighted scoring, timed-mode bonus, failure penalty.

### 5.3 End of Game

The game ends when all triples have been found.

- Winner: the player with the highest score.
- Tie: shared win in MVP; tie-break round deferred.

## 6. Data Model

The game separates **semantic objects** (reference database) from **display cards** (gameplay deck).

### 6.1 Entity Model

```json
{
  "country_id": "AUT",
  "country_name": "Austria",
  "capital_name": "Vienna",
  "alternative_capitals": [],
  "river_name": "Danube",
  "shared_river_flag": true,
  "shared_river_countries": ["DEU", "SVK", "HUN", "HRV", "SRB", "ROU", "BGR", "MDA", "UKR"],
  "selection_reason": "major_historic",
  "capital_selection_reason": "sole_official",
  "provenance": "CIA World Factbook 2025; UN member state list",
  "recognition_status": "un_member",
  "difficulty": 1,
  "region": "Europe",
  "endorsement_marker": "internal_curation_v0.3",
  "is_active": true
}
```

**v0.3 note on provenance:** the `provenance` field is required, but at MVP it may be as short as a single source reference (e.g. `"UN member list; standard atlas"`). Longer provenance entries are expected as the deck grows; minimal-but-honest is acceptable at launch.

### 6.2 Card Model

```json
{
  "card_id": "card_001",
  "country_id": "AUT",
  "mode_tags": ["same_country"],
  "card_type": "capital",
  "label": "Vienna",
  "label_script": "latin",
  "initial_letter": "V",
  "romanisation_source": "native_latin"
}
```

### 6.3 Romanisation Rule

For the MVP, all card labels are stored and displayed in their conventional English-language Latin-script form. The `initial_letter` field is derived from the displayed Latin-script label. A future version may add a script-switching toggle, at which point letter-matching must be re-defined per script.

### 6.4 Suggested Fields — consolidated

**Country-level:** `country_id`, `country_name`, `capital_name`, `alternative_capitals`, `river_name`, `shared_river_flag`, `shared_river_countries`, `selection_reason`, `capital_selection_reason`, `provenance`, `recognition_status`, `difficulty`, `region`, `endorsement_marker`, `is_active`.

**Card-level:** `card_id`, `country_id`, `card_type`, `label`, `label_script`, `initial_letter`, `romanisation_source`, `mode_tags`.

## 7. River Selection Policy

A river is never chosen arbitrarily. Each country entry must include a canonical `river_name`, a `selection_reason` tag, a `shared_river_flag`, `provenance`, and an `endorsement_marker`.

**Permitted `selection_reason` values:** `longest_in_country`, `largest_basin`, `capital_region`, `major_historic`, `iconic`, `cross_border_reference`, `none`.

**Canonical Rule:**

> The game uses one canonical river per country entry as a gameplay convention. This is a curation choice, not a geographic law. For countries with multiple plausible rivers, the canonical choice is documented in the Curation Statement (§18) with a brief justification.

### 7a. Capital Selection Policy

For countries with more than one capital (South Africa, Netherlands, Bolivia, Eswatini) or contested capitals (Israel/Palestine):

- a canonical `capital_name` (used in gameplay),
- an `alternative_capitals` list,
- a `capital_selection_reason` from: `sole_official`, `executive`, `legislative`, `judicial`, `historic`, `de_facto`, `contested_documented`.

### 7b. Country Inclusion and Recognition Policy

Each entry carries a `recognition_status`:

- `un_member`
- `un_observer`
- `widely_recognised`
- `limited_recognition`
- `dependency`

The MVP deck is restricted to `un_member` entries. Broader tiers are enabled only via teacher opt-in in later phases, with a disclaimer surface.

## 8. Dataset Strategy

### 8.1 Phase 1 — Manual Curated Core

- 20 curated country triples (Same Country Mode).
- 20 curated letter groups (Same Letter Mode).
- All entries UN-member states for MVP.
- Each entry validated against §7 / §7a / §7b.

The MVP round draws 4–8 triples from this pool.

### 8.2 Phase 2 — Expanded Database

- Expand country coverage.
- Refine letter balance; introduce cross-continental balance constraints if playtest data justifies them (see §9.2).
- Introduce multiple difficulty levels at scale.
- Optional: enable `widely_recognised` recognition tier via teacher toggle.

### 8.3 Phase 3 — Full Library

Global deck, regional decks, children's mode, expert mode, language switching (triggers §6.3 revision).

### 8.4 Phase Gates

A deck progresses between phases only when:

- every entry has non-empty `provenance`,
- every entry has an `endorsement_marker`,
- the Curation Statement has been updated,
- at least one playtest session with the target audience has been logged.

## 9. Same Letter Mode Curation Strategy

Letter Mode uses curated playable triples, not auto-generated combinations.

### 9.1 Letter Group Schema

```json
{
  "letter_group_id": "B_01",
  "initial_letter": "B",
  "cards": [
    { "card_type": "capital", "label": "Berlin", "country_id": "DEU" },
    { "card_type": "country", "label": "Belgium", "country_id": "BEL" },
    { "card_type": "river", "label": "Brahmaputra", "country_id": "IND" }
  ],
  "difficulty": 2,
  "endorsement_marker": "internal_curation_v0.3",
  "is_active": true
}
```

**Schema constraint:** every letter group must contain exactly one card of each `card_type`. Groups that fail this validation are rejected at load time.

### 9.2 Curation Filter Protocol [revised, v0.3]

A letter group is accepted into the Phase 1 pool only if it meets:

- **Familiarity floor** — each of the three labels is present in a reference list of the most commonly taught geographic names (school-atlas level).
- **Type coverage** — exactly one capital, one country, one river (enforced by §9.1).
- **No false-pattern trap** — a group must not appear to form a geographic pattern that it does not. For example, "Berlin, Bavaria, Bodensee" would be rejected because all three are German, inviting a geographic read of what is meant to be a purely lexical triple.

**Removed in v0.3:** the cross-continental balance cap has been deferred to Phase 2. The external critique correctly observed that enforcing this at MVP rejects otherwise strong groups. Phase 2 may re-introduce it if playtesting shows that uncapped cross-continental spread causes player confusion.

### 9.3 Board-level Constraint [revised, v0.3]

Within a single round, the board generator **prefers** to avoid configurations where two different triples place cards of the same `card_type` with the same `initial_letter` (e.g. Paris and Prague, both capitals starting with P, from two different letter groups).

**Softening from v0.2:** this is a preference, not a prohibition. If the candidate pool is small enough that avoidance would make board generation brittle, the generator may proceed and **log the collision** for playtest review. After initial playtesting, this may harden into a rule, be loosened further, or be removed.

The external critique flagged this as over-constraint; the empirical evidence from playtesting will adjudicate.

## 10. Gameplay Architecture

### 10.1 First Technical Target

A browser-based game running locally on one screen.

**Suggested stack:** HTML, CSS, JavaScript (React optional). No backend for v0.1 beyond local JSON loading. JSON file storage. In-browser state management.

### 10.2 Game States

1. Start screen.
2. Mode selection (with progression staging per §4a).
3. Player count selection.
4. Difficulty selection.
5. Board generation.
6. Active game.
7. End screen with score summary.
8. Restart or return to menu.

### 10.3 Board Generation

**Same Country Mode:** choose N country entries from the active pool; create 3 cards per entry; shuffle; place on board.

**Same Letter Mode:** choose N curated letter groups, applying §9.3 collision preference; create 3 cards per group; shuffle; place on board.

### 10.4 Tutorial Mode [new, v0.3]

The MVP includes a lightweight tutorial to introduce the game logic before the first full round.

**Goals:**

- Explain the three card types (capital, country, river).
- Demonstrate the difference between Same Country and Same Letter.
- Show that valid matches are triples, not pairs.
- Reduce first-contact rule confusion, especially for children and casual players.

**Structure (three steps):**

**Step 1 — Card-type introduction.**
Three revealed example cards are displayed with their type icons and labels clearly shown:

- Capital: Vienna
- Country: Austria
- River: Danube

A short caption explains that every triple contains one of each type.

**Step 2 — One guided example per mode.**
Same Country example: the player is shown six revealed cards (two triples) and asked to click the three that form the Vienna · Austria · Danube triple. A correct click rewards; an incorrect click gently highlights the correct answer.

Same Letter example: the player is shown six revealed cards across two P-groups (e.g. Prague · Portugal · Po plus Paris · Peru · Parana) and asked to click any valid P-triple. Both triples are acceptable answers.

**Step 3 — One guided mini-round.**
A real but small board with 2–3 triples (6–9 cards), face-down. Mode is clearly labelled; card types are visible on reveal. Optional soft hints after a wrong triple attempt (e.g. highlighting one correct card that was already revealed).

After Step 3, the player enters the real game.

**MVP constraints:**

- No narration, dialogue, or multi-level tutorial flow.
- No characters or mascots.
- Optional from the main menu; automatically offered on first launch with a skip option.
- The tutorial uses the same visual design as the main game — no separate art style.

**Rationale:** the tutorial is an ethics-preserving addition. A player who understands the rules before playing engages honestly; a confused player learns only to guess, which contradicts §2.

## 11. UI Principles

The interface is designed for a shared screen.

**Requirements:**

- Large readable text.
- Strong visual distinction between hidden and revealed cards.
- Visible current player.
- Visible score.
- Visible mode.
- **Card type visible by default** on revealed cards (small icon or word: *Capital / Country / River*).

**Expert mode:** type-hiding is available as a difficulty toggle (§12), not as the default presentation.

### 11a. Accessibility Baseline

The MVP must meet, at minimum:

- **Contrast:** WCAG 2.2 AA for all text on cards and UI chrome.
- **Font size:** no gameplay text below 18px at default zoom.
- **Colour-only signalling:** prohibited. Any state conveyed by colour must also be conveyed by shape, icon, text, or pattern.
- **Keyboard navigation:** all card reveals, mode selections, and menu actions reachable via Tab and activatable via Enter/Space.
- **Screen reader:** each card has an `aria-label` reflecting its type and label when revealed; hidden cards announce "hidden card, position N".
- **Reduced motion:** card-flip animations honour `prefers-reduced-motion`.

These are floor requirements, not targets. Guardian holds this section against any MVP scope pressure.

## 14. Risks and Edge Cases

### 14.1 Country Mode ambiguity

Not every country has one obvious "signature river". Mitigation: documented canonical choice (§7) plus Curation Statement (§18).

### 14.2 Letter Mode imbalance

Some letters are over-represented, others weak. Mitigation: §9.2 curation filter.

### 14.3 Educational tension

Too much rigour reduces fun; too much looseness reduces trust. Mitigation: gameplay deck and reference database remain separated (§6); v0.3 reduces governance overhead at MVP (§15).

### 14.4 Political recognition

Including or omitting a polity is political. Mitigation: `recognition_status` field; UN-member-only MVP (§7b).

### 14.5 Script and romanisation

Letter Mode is undefined across scripts. Mitigation: §6.3 romanisation rule; Latin-script gameplay for MVP.

### 14.6 Ambiguous labels

Some labels are legitimately ambiguous (e.g. "Jordan" — country and river). Mitigation: visible `card_type` in default mode (§11); disambiguated labels in data where helpful ("Jordan River" for the river card).

### 14.7 UNDERDETERMINED match state

Deferred: if future rulesets allow a card to participate in more than one valid triple simultaneously, the engine must return an UNDERDETERMINED result with a Discriminant Condition. MVP avoids this by single-mode rounds and fixed triples.

### 14.8 Mode conflation in player perception [new, v0.3]

Players may misread Same Letter Mode as a geography-reference mode. Mitigation: §3.4 asymmetry statement; Curation Statement makes the distinction explicit; Tutorial Mode demonstrates both modes side by side.

### 14.9 Over-governance [new, v0.3]

The curation scaffolding may visually or cognitively dominate play. Mitigation: §23 Governance Visibility Principle.

## 17. Dispute and Feedback Channel [revised, v0.3]

For MVP, the dispute channel is a single line in `README.md`:

> **Corrections and disputes:** if you believe an entry is wrong, misleading, or culturally inappropriate, please open an issue in the repository or contact the maintainers at [address]. Corrections are reviewed and recorded in the Curation Statement.

A full `DISPUTES.md` with formal process, response windows, and archived trails is deferred to Phase 2, when deck size and user base make the formal process useful.

This follows Guardian principle: the channel exists from day one (non-negotiable) but is not over-engineered before there is traffic to justify it.

## 18. Curation Statement [revised, v0.3]

Every published deck is accompanied by `CURATION_STATEMENT.md`. At MVP, this document may be **partial**:

- At launch, only non-trivial canonical choices need to be documented. Obvious cases (Paris as capital of France; the Nile for Egypt) do not require entries.
- Each non-obvious choice (Seine vs. Loire; multi-capital states; contested political recognition as encountered) receives a short paragraph.
- The statement grows organically as the deck expands.

**Required at MVP:**

- A one-paragraph preamble explaining the Harbour-style distinction between "our gameplay canon" and "geographic fact".
- At least three worked examples of non-trivial curation choices, one each for river, capital, and inclusion.
- The provenance source list (may be brief).
- The current `endorsement_marker` version.

**Optional (may grow later):**

- Full entry-by-entry commentary.
- Historical endorsement-marker trail.
- Teacher-facing lesson suggestions.

The statement is linked from the main menu footer (§23), not shown during gameplay.

## 22. Short Position

v0.2 was honest but over-built for its first playable target. v0.3 preserves every ethical commitment — type visibility, accessibility, honest curation, dispute access, canonical-choice transparency — while relieving the MVP of scaffolding it does not yet need.

The principal addition is the **Tutorial Mode**, which is better ethics than more metadata: it turns a potentially confusing first encounter into a structured introduction. The principal softenings (continental cap, collision rule, dispute process, Curation Statement completeness) move from **prescribed** to **observed** — the playtest, not the blueprint, decides where they harden.

The Lock-Key Rule holds. Concepts remain stable; interpretations remain declared. The declarations are now allowed to grow at a sensible pace rather than being fully specified before the first card is flipped.

## 23. Governance Visibility Principle [new, v0.3]

Governance serves play, not the reverse.

**Rules:**

- The Curation Statement is linked from the **main menu footer** or from a **Settings → About** screen. It is never shown during gameplay.
- The dispute contact line lives in the **README** and, optionally, in the same footer/about location. It is never surfaced on the playing surface.
- The `endorsement_marker` on data entries is machine-readable; it is not displayed on cards, scoreboard, or end screen.
- `provenance` and `recognition_status` fields are not exposed in-game at MVP. They may be surfaced in a future "teacher view" that is explicitly opt-in.

**Rationale:** v0.2's correct commitment to transparency risked visual imposition. A child or casual player should experience a clean card game, with the governance layer reachable when sought but invisible when not.

This section operationalises the third design principle in §2: *Governance serves play.*
