## 1. Concept

This game combines elements of **Memory** and **City–Country–River** into a structured card-matching game.

Each logical set consists of **three linked cards**:

- **Capital**
- **Country**
- **River**

Example:

- Paris
- France
- Loire

A successful match is not a pair, but a **triple**.

The game supports two modes:

1. **Same Country Mode** — three cards belong to the same country.
2. **Same Letter Mode** — three cards share the same initial letter.

The first implementation is a **local multiplayer game on one screen**, with players taking turns.

---

## 2. Core Design Goal [revised]

The game should be:

- easy to understand,
- visually clear on a single monitor,
- **educationally transparent** — players should always be able to tell what kind of entity a card represents by default,
- flexible enough to support curated datasets and later expansion.

The primary design principle is:

> The data model must be clean enough that gameplay remains unambiguous, and the curation must be honest enough that players can trust what they are learning.

A secondary principle, added in v0.2:

> Opacity is a difficulty setting, not a default. Hiding card type belongs in an expert mode, not the base experience.

---

## 3. Modes

### 3.1 Same Country Mode

**Rule:** a valid set consists of the capital of a country, the country name, and a designated river associated with that country.

**Example:** Vienna · Austria · Danube

**Matching logic:** a triple is correct if all three cards share the same `country_id`.

**Advantage:** semantically strong and educational.

**Challenge:** the river must be curated carefully. "Most important river" is not always objectively defined — see §7.

### 3.2 Same Letter Mode

**Rule:** a valid set consists of three cards whose labels begin with the same initial letter, **and** cover the three entity types (one Capital, one Country, one River). The type-coverage constraint is new in v0.2 and was implicit in v0.1.

**Example:** Berlin · Belgium · Brahmaputra

**Matching logic:** a triple is correct if all three cards share the same `initial_letter` **and** the triple contains exactly one card of each `card_type`.

**Advantage:** simpler, faster, exercises phonetic and lexical memory rather than geographic memory.

**Challenge:** less geographically rigorous. The dataset must be curated; see §9.

### 3.3 Mode Exclusivity [new]

Within a single round, only one mode is active. Cards on the board are drawn from one curated deck (Same Country or Same Letter), never mixed. A triple that would coincidentally satisfy both modes (e.g. Bern, Belarus, Brahmaputra — all B, none geographically linked) is not possible within a single round, because the active deck governs which triples exist.

This resolves the **dual-mode collision** signal (Scout, §4).

## 4. Recommended First Version

For the first playable prototype:

- implement **both modes**,
- start with a **small curated dataset**,
- keep the number of cards limited,
- prioritise clarity over scale.

Recommended initial board sizes:

- **12 cards** = 4 triples (children or introductory play)
- **18 cards** = 6 triples (**default MVP**)
- **24 cards** = 8 triples (standard play)

**Clarification [new]:** these board sizes refer to the **number of cards drawn per round**. They are sampled from a larger curated pool. The Phase 1 pool is 20 triples per mode (§8.1), from which 4–8 triples are selected per round.

## 5. Player Experience

### 5.1 Turn Sequence

A player reveals cards one by one, up to three cards per turn.

#### Option A — Strict Triple Turn (selected for MVP)

- Reveal card 1.
- Reveal card 2.
- Reveal card 3.
- System checks whether the three cards form a valid triple.

If correct: cards remain open; player scores one point; **player's turn ends** (see §5.1.1).

If incorrect: cards flip back after a short delay; next player takes over.

#### Option B — Early Failure Logic

Deferred to a later version.

#### 5.1.1 Turn Continuation Rule [new]

v0.1 left "continues or ends turn depending on rule setting" unresolved. v0.2 resolves this for the MVP:

> **MVP rule:** after a correct triple, the player's turn **ends**. Play passes to the next player regardless of outcome.

Rationale: this keeps session length predictable, ensures all players remain engaged between turns, and matches the pedagogical symmetry of classical Memory. A "keep the turn on success" variant is documented as a future option but not implemented in v0.1 of the code.

This resolves the **turn continuation occlusion** signal (Scout, §4).

### 5.2 Scoring

Base scoring: **1 point** per correct triple.

Optional later additions (not in MVP):

- bonus for consecutive correct triples,
- difficulty-weighted scoring,
- timed-mode bonus,
- penalty for failed attempts.

### 5.3 End of Game

The game ends when all triples have been found.

- Winner: the player with the highest score.
- Tie: shared win in MVP; a tie-break round is deferred.

## 6. Data Model

The game separates **semantic objects** (reference database) from **display cards** (gameplay deck). This layering mirrors the Harbour distinction between invariant coastlines and navigational practice.

### 6.1 Entity Model [revised]

Each country-related entry now contains provenance and status fields.

```json
{
  "country_id": "AUT",
  "country_name": "Austria",
  "capital_name": "Vienna",
  "alternative_capitals": [],
  "river_name": "Danube",
  "shared_river_flag": true,
  "shared_river_countries": ["DEU", "SVK", "HUN", "HRV", "SRB", "ROU", "BGR", "MDA", "UKR"],
  "selection_reason": "major_historic",
  "capital_selection_reason": "sole_official",
  "provenance": "CIA World Factbook 2025; UN member state list",
  "recognition_status": "un_member",
  "difficulty": 1,
  "region": "Europe",
  "endorsement_marker": "internal_curation_v0.2",
  "is_active": true
}
```

New fields (v0.2):

- `alternative_capitals` — empty list, or the non-canonical capitals for countries with more than one (e.g. Netherlands, South Africa, Bolivia).
- `shared_river_flag` (boolean) — true if the river flows through multiple countries.
- `shared_river_countries` — ISO codes for other riparian states, if `shared_river_flag` is true.
- `capital_selection_reason` — see §7a.
- `provenance` — a source citation for the entry (see §7, §18).
- `recognition_status` — see §7b.
- `endorsement_marker` — identifies the curation layer, separating "our gameplay canon" from "externally validated fact".

### 6.2 Card Model [revised]

```json
{
  "card_id": "card_001",
  "country_id": "AUT",
  "mode_tags": ["same_country"],
  "card_type": "capital",
  "label": "Vienna",
  "label_script": "latin",
  "initial_letter": "V",
  "romanisation_source": "native_latin"
}
```

New fields (v0.2):

- `label_script` — the script of the displayed label (`latin`, `cyrillic`, `cjk`, etc.).
- `romanisation_source` — see §6.3.

### 6.3 Romanisation Rule [new]

v0.1 left Letter Mode matching undefined for non-Latin-script labels (Moscow vs. Москва; Kyiv vs. Київ; Seoul vs. 서울).

v0.2 rule:

> For the MVP, all card labels are stored and displayed in their conventional English-language Latin-script form. The `initial_letter` field is derived from the displayed Latin-script label. A future version may add a script-switching toggle, at which point letter-matching must be re-defined per script.

`romanisation_source` values: `native_latin`, `standard_english`, `bgn_pcgn`, `pinyin`, `revised_romanisation`, `iso_9`, etc. The MVP uses `standard_english` as the default gameplay convention.

### 6.4 Suggested Fields — consolidated

**Country-level:** `country_id`, `country_name`, `capital_name`, `alternative_capitals`, `river_name`, `shared_river_flag`, `shared_river_countries`, `selection_reason`, `capital_selection_reason`, `provenance`, `recognition_status`, `difficulty`, `region`, `endorsement_marker`, `is_active`.

**Card-level:** `card_id`, `country_id`, `card_type`, `label`, `label_script`, `initial_letter`, `romanisation_source`, `mode_tags`.

## 7. River Selection Policy [revised]

A river is never chosen arbitrarily. Each country entry must include:

1. a canonical `river_name`,
2. a `selection_reason` tag,
3. a `shared_river_flag`,
4. `provenance` (source of the choice),
5. an `endorsement_marker` declaring the curation layer.

**Permitted `selection_reason` values:**

- `longest_in_country`
- `largest_basin`
- `capital_region`
- `major_historic`
- `iconic`
- `cross_border_reference`
- `none`

**Canonical Rule [revised wording]:**

> The game uses **one canonical river per country entry** as a gameplay convention. This is a curation choice, not a geographic law. For countries with multiple plausible rivers, the canonical choice is documented in the Curation Statement (§18) with a brief justification. Players and teachers are invited to consult the Curation Statement for context.

Example:

```json
{
  "country_id": "EGY",
  "country_name": "Egypt",
  "capital_name": "Cairo",
  "river_name": "Nile",
  "selection_reason": "iconic",
  "shared_river_flag": true,
  "provenance": "Consensus usage; UN FAO Aquastat",
  "endorsement_marker": "internal_curation_v0.2"
}
```

This resolves the **river authority gap** signal (Scout, §4) and the unmarked-canonical-river concern (Guardian).

### 7a. Capital Selection Policy [new]

For countries with more than one capital (South Africa, Netherlands, Bolivia, Eswatini, etc.) or contested capitals (Israel/Palestine, Jerusalem status), the entry must specify:

- a canonical `capital_name` (used in gameplay),
- an `alternative_capitals` list,
- a `capital_selection_reason` from: `sole_official`, `executive`, `legislative`, `judicial`, `historic`, `de_facto`, `contested_documented`.

The Curation Statement (§18) records the reasoning for each non-trivial case.

### 7b. Country Inclusion and Recognition Policy [new]

Including or omitting a polity is a political act. v0.2 requires each entry to carry a `recognition_status` field:

- `un_member` — UN member state.
- `un_observer` — UN observer (e.g. Vatican, Palestine).
- `widely_recognised` — recognised by a majority of UN members but not a UN member (e.g. Kosovo, Taiwan, depending on framing).
- `limited_recognition` — recognised by some states (e.g. Western Sahara, Northern Cyprus).
- `dependency` — non-sovereign territory included for educational reasons.

The MVP deck is restricted to `un_member` entries to minimise controversy at launch. Broader categories may be enabled via an "expanded recognition deck" in later phases, with explicit teacher opt-in and a disclaimer surface.

This addresses the political-recognition risk raised by the Guardian.

## 8. Dataset Strategy

### 8.1 Phase 1 — Manual Curated Core

- 20 curated country triples (Same Country Mode).
- 20 curated letter groups (Same Letter Mode).
- All entries UN-member states for MVP.
- Each entry validated against the §7 / §7a / §7b policies.

The MVP round draws 4–8 triples from this pool; the pool is larger than any single round to enable replayability.

### 8.2 Phase 2 — Expanded Database

- Expand to more countries.
- Refine letter balance.
- Introduce multiple difficulty levels.
- Optional: enable `widely_recognised` recognition tier via teacher toggle.

### 8.3 Phase 3 — Full Library

- Global deck.
- Regional decks.
- Children's mode.
- Expert mode.
- Language switching (triggers romanisation rule revision per §6.3).

### 8.4 Phase Gates [new]

A deck progresses between phases only when:

- every entry has a non-empty `provenance`,
- every entry has an `endorsement_marker`,
- the Curation Statement (§18) has been updated,
- at least one playtest session with the target audience has been logged.

## 9. Same Letter Mode Curation Strategy [revised]

Letter Mode uses curated playable triples, not auto-generated combinations.

### 9.1 Letter Group Schema

```json
{
  "letter_group_id": "B_01",
  "initial_letter": "B",
  "cards": [
    { "card_type": "capital", "label": "Berlin", "country_id": "DEU" },
    { "card_type": "country", "label": "Belgium", "country_id": "BEL" },
    { "card_type": "river", "label": "Brahmaputra", "country_id": "IND" }
  ],
  "difficulty": 2,
  "endorsement_marker": "internal_curation_v0.2",
  "is_active": true
}
```

**Schema constraint (Architect):** every letter group must contain **exactly one** card of each `card_type` (capital, country, river). This is enforced at dataset load time; groups that fail validation are rejected.

### 9.2 Curation Filter Protocol [new]

A letter group is accepted into the Phase 1 pool only if it meets:

- **Familiarity floor** — each of the three labels is present in a reference list of 500 most commonly taught geographic names (school-atlas level).
- **Cross-continental balance cap** — no more than one continent may be represented per group, or all three must be on distinct continents (both are acceptable; "two of three" is rejected as cognitively misleading).
- **No false-pattern trap** — a group must not appear to form a geographic pattern that it does not. For example, "Berlin, Bavaria, Bodensee" would be rejected because all three are German, inviting a geographic read of what is meant to be a purely lexical triple.
- **Letter uniqueness per round** — the board generator must not place two active letter groups whose constituent labels share the same letter (trivially all do for the group letter, but see §9.3).

This resolves the **Letter Mode quality control** signal (Scout, §4).

### 9.3 Board-level Constraint [new, Architect]

Within a single round, the board must not contain two cards of the same `card_type` sharing the same `initial_letter` across different triples.

Example of a forbidden configuration: a Same Letter round with Paris (capital, P) from one triple and Prague (capital, P) from another. Even though the letter groups are distinct (e.g. the P group and some other group), the coincidence would force players into guessing-game territory.

The board generator sorts candidate groups and rejects any sampling that produces such a collision.

## 10. Gameplay Architecture

### 10.1 First Technical Target

A browser-based game running locally on one screen.

**Suggested stack:**

- Frontend: HTML, CSS, JavaScript (React optional).
- Backend: none for v0.1, or minimal local JSON loading.
- Data storage: JSON files.
- State management: in-browser game state.

### 10.2 Game States

1. Start screen.
2. Mode selection.
3. Player count selection.
4. Difficulty selection (new in v0.2, see §12).
5. Board generation.
6. Active game.
7. End screen with score summary.
8. Restart or return to menu.

### 10.3 Board Generation

**Same Country Mode:**

- Choose N country entries from the active pool.
- Create 3 cards per entry.
- Shuffle all cards.
- Place on board.

**Same Letter Mode:**

- Choose N curated letter groups, applying §9.3 letter-collision check.
- Create 3 cards per group.
- Shuffle all cards.
- Place on board.

## 11. UI Principles [revised]

The interface is designed for a shared screen.

**Requirements:**

- Large readable text.
- Strong visual distinction between hidden and revealed cards.
- Visible current player.
- Visible score.
- Visible mode.
- **[revised]** Card type is **visible by default** on revealed cards (e.g. a small icon: 🏛️ capital, 🗺️ country, 🌊 river, or the words *Capital / Country / River*).

**Expert mode:** type-hiding is available as a difficulty toggle (§12), not as the default presentation.

Rationale: v0.1 suggested hiding type "may make the game more interesting". This undermines the educational goal (§2). A player who does not know whether "Jordan" is a country or a river cannot learn geography from the game — they learn only to guess. Visible type turns the guessing game into a memory-and-categorisation game, which is the intended design.

### 11a. Accessibility Baseline [new]

The MVP must meet, at minimum:

- **Contrast:** WCAG 2.2 AA for all text on cards and UI chrome.
- **Font size:** no gameplay text below 18px at default zoom.
- **Colour-only signalling:** prohibited. Any state conveyed by colour must also be conveyed by shape, icon, text, or pattern.
- **Keyboard navigation:** all card reveals, mode selections, and menu actions are reachable via Tab and activatable via Enter/Space.
- **Screen reader:** each card has an `aria-label` reflecting its type and label when revealed; hidden cards announce "hidden card, position N".
- **Reduced motion:** card-flip animations honour `prefers-reduced-motion`.

These are floor requirements, not targets.

## 14. Risks and Edge Cases [revised]

### 14.1 Country Mode ambiguity

Not every country has one obvious "signature river". Mitigation: documented canonical choice (§7) plus Curation Statement (§18).

### 14.2 Letter Mode imbalance

Some letters are over-represented, others weak. Mitigation: §9.2 curation filter.

### 14.3 Educational tension

Too much rigour reduces fun; too much looseness reduces trust. Mitigation: gameplay deck and reference database remain separated (§6).

### 14.4 Political recognition [new]

Including or omitting a polity is political. Mitigation: `recognition_status` field and UN-member-only MVP (§7b).

### 14.5 Script and romanisation [new]

Letter Mode is undefined across scripts. Mitigation: §6.3 romanisation rule; Latin-script gameplay for MVP.

### 14.6 Ambiguous labels [new]

Some labels (e.g. "Jordan" — country and river; "Mexico" — country and capital nickname) are legitimately ambiguous. Mitigation: visible `card_type` in default mode (§11); labels disambiguated in data where possible ("Jordan River" rather than "Jordan" for the river card).

### 14.7 UNDERDETERMINED match state [new]

If a future ruleset allows a card to participate in more than one valid triple simultaneously (not permitted in MVP), the engine must return an UNDERDETERMINED result with a Discriminant Condition stating which interpretation the player intends. The MVP avoids this by single-mode rounds and fixed triples.

## 15. Minimal Viable Product (MVP) [revised]

The MVP includes:

- Mode selection (Same Country, Same Letter).
- Difficulty selection (Levels 1–3; Level 4 off by default).
- 2–4-player local mode.
- **6 playable triples per round (18 cards)** — resolved as the default.
- **Turn continuation: turn ends after each triple attempt** (§5.1.1).
- **Card type visible by default** (§11).
- Accessibility baseline met (§11a).
- Score tracking.
- Turn handling.
- Card flip animation (with reduced-motion honoured).
- End screen.
- Curation Statement published alongside the deck (§18).
- Dispute channel reachable from the main menu (§17).

Not in MVP:

- Online multiplayer.
- User accounts.
- Database server.
- Full-world dataset.
- Timers.
- Sound effects.
- Expert-mode type hiding (available but off by default).
- Non-Latin script support.

## 17. Dispute and Feedback Channel [new]

Every published deck carries a `DISPUTES.md` document in the repository. The document states:

- how a player, teacher, or reader may raise a correction,
- the expected response window,
- how accepted corrections are recorded in the `endorsement_marker` history,
- that disputes are archived even if not acted upon, preserving the curation trail.

This is the Breakwater-compatible channel: corrections enter as candidate claims, not immediate overrides.

---

## 18. Curation Statement [new]

Every published deck is accompanied by `CURATION_STATEMENT.md`, which records:

- Why each canonical river was chosen over alternatives.
- Why each canonical capital was chosen (for multi-capital states).
- Which countries were included or excluded under the recognition policy.
- The provenance source list.
- The `endorsement_marker` version and date.
- Known limitations and open questions.

The statement is intended for teachers, parents, and older players. It is not shown during gameplay but is linked from the main menu.

This directly addresses the Guardian's concern that v0.1 locked interpretations silently while leaving concepts drifting. The Curation Statement inverts this: concepts remain concepts, interpretations are declared.

## 20. Design Decision Summary [revised]

**Keep:**

- Triple-based matching.
- Two clearly separated modes.
- Curated datasets.
- Local multiplayer first.
- Browser-based prototype.
- Semantic / display separation.

**Add in v0.2:**

- Provenance and recognition metadata on every entry.
- Canonical choices declared, not hidden.
- Accessibility baseline.
- Dispute channel.
- Curation Statement.
- Type-visible default.

**Avoid:**

- Treating curation as editorial rather than epistemic work.
- Using a full global dataset too early.
- Mixing strict geography and playful letter logic in one unclear rule layer.
- Hiding card type as a default.
- Colour-only signalling.

## 22. Short Position

The v0.1 concept was sound. v0.2 does not change the game; it makes the curation honest.

The blueprint is harbourworthy: the data model is a clean coastline, the modes exercise orthogonal memory systems, and the MVP scope is deliberately small. The v0.2 additions — provenance, recognition status, Curation Statement, accessibility floor, default type visibility, and a dispute channel — bring the project into alignment with the Lock-Key Rule: the concepts (country, capital, river) stay as stable locks; the interpretations (which river, which capital, which country) are declared as keys, visible to the player and reviewable by teachers.

If the data layer is well designed and the curation is transparently documented, the implementation is straightforward.
